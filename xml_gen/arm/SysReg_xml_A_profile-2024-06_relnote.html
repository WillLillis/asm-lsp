Introduction

This is the 2024-06 release of the System Register XML for A-profile Architecture, describing:

  • The AArch64 and AArch32 views of the System registers, including Debug, PMU, AMU, MPAM, Generic Timer, and GIC.
  • The AArch64 and AArch32 system control operations.
  • The memory-mapped Debug, CTI, PMU, AMU, MPAM, GIC, and Generic Timer registers.

The Proprietary Notice gives details of the terms and conditions under which this package is provided.

If you have comments on the content of this package, create a ticket at https://support.developer.arm.com. As part of the ticket, include:

  • The title, "System Register XML for A-profile Architecture".
  • The version, "2024-06".
  • The section name to which your comments refer.
  • A concise explanation of your comments.

Please see the Documentation for more information on the general structure of these descriptions.

Product Status

The information in this release covers multiple versions of the architecture. The content relating to different versions is given different quality ratings.

The information relating to the 2023 Extensions and the rest of the A-profile Architecture is at Beta quality. Beta quality means that all major features of the specification are described, but some details might be missing.

Change history

  • The feature dependences for FEAT_SSVE_FP8FMA and FEAT_SSVE_FP8DOT2 in ID_AA64SMFR0_EL1 are relaxed to no longer require corresponding ZA-targeting FP8 instructions.
  • DBGDTRTX_EL0 is corrected to remove the reference to DTRRX.
  • SCTLR2_EL2.EnANERR and SCTLR2_EL2.EnADERR are extended to include the explicit memory accesses generated as a result of an MRS/MRRS register access being transformed to a memory access by configuration of HCR_EL2.NV2=1.
  • PMEVTYPER<n>_EL0, PMCCFILTR_EL0 and PMICFILTR_EL0 resets in the upper 32 bits of the registers are updated to be 0 if AArch32 is available.
  • A new backwards-compatible field "NSMC, bits [59:56]" is added to SMIDR_EL1.
  • In TRBPTR_EL1, instances of PMBIDR_EL1 are corrected TRBIDR_EL1.
  • TRBSR_EL1.DAT is relaxed in the AArch64 version of the register to be UNKNOWN when FEAT_TRBE_EXT is implemented, and RES0 otherwise.
  • The EL0 and EL1 read conditions in CNTVCT_EL0.Bits and CNTVCTSS_EL0.Bits are corrected.
  • EDPRSR field accessors are updated to have consistent UNKNOWN/WI access if FEAT_DoPD is not implemented, or the Core domain is not powered up, or the PE is in reset state.
  • CTIDEVARCH.REVISION is clarified to say that the value 0b0000 is not permitted when FEAT_DoPD is implemented.
  • The list of instruction affected by the SCTLR_ELx.BTx bits are updated to include the PACIASPPC and PACIBSPPC instructions.
  • The SME non-widening BFloat16 instructions are moved out of FEAT_SVE_B16B16 and into a new feature FEAT_SME_B16B16.
  • SSBS.SBSS and CPSR.SBSS descriptions of speculation are clarified using the term speculative register values.
  • GIC*_IPRIORITYR*.Priority_offset_x descriptions are updated to indicate they are RES0 for an INTID configured as non-maskable.
  • Constraints on ID_AA46DFR0_EL1.TraceFilt are updated to indicate permitted values for ID_AA46DFR0_EL1.TraceFilt when FEAT_ETE is implemented.
  • Performance Monitors registers with event fields that have conditional accessors when fewer than 31 event counters are implemented are replaced with conditions that use the GetNumEventCounters() function.
  • The EC syndrome value for trapped AArch32 accesses due to HCR_EL2.TPU is corrected to 0x03.
  • GICR_VPENDBASER.Dirty is corrected to remove the reserved clause.
  • ID_AA64PFR0_EL1 is clarified to refer to out of context prediction resources.
  • The description of ICH_LR<n>.State is clarified to include description of behaviour when a virtual interrupt from a List register is acknowledged.
  • The incorrect reference to traps in CNTHCTL_EL2.ECV 0b1 value is removed.
  • External *DEVAFF registers have been updated to match the equivalent content in MPIDR_EL1.
  • The field descriptions in GICR_VPROPBASER are corrected to refer to the vPE Configuration Table.
  • Correction to purpose description of PMSELR_EL0 to refer to PMEVCNTR<n>_EL0 and PMCCNTR_EL0.
  • Configuration text of External registers MPAMCFG_CASSOC, MPAMCFG_CMIN, MPAMCFG_DIS, MPAMCFG_EN, and MPAMCFG_EN_FLAGS is updated to clarify the power and reset domains of MSC components.
  • External debug register descriptions with respect to RXfull or TXfull are clarified to state the read or write behaviour is 0 or 1.
  • System registers describing the LPA2 features changes to ID_AA64MMFR0_EL1, {V}TCR_ELx, and {V}TTBRy_ELx are corrected.
  • The DS field in TCR_EL1, TCR_EL2, VTCR_EL2 and TCR_EL3 affects both input and output addresses.
  • FPCR.DN is corrected to have no effect on the output of the FABS and FNEG instructions and to have no effect on the following instructions when FPCR.AH is 1:
    • FMAX.
    • FMAXP.
    • FMAXV.
    • FMIN.
    • FMINP.
    • FMINV.
  • HCRX_EL2 behavior when EL2 is not implemented or SCR_EL3.HXEn is 0, and the interaction of the value of HCRX_EL2.MSCEn is clarified.
  • The Effective value of RW in SCR_EL3 is corrected to consider the scenario where EL2 is not implemented.
  • ACTLR_EL12 and IMPLEMENTATION DEFINED accesses for ACTLR_EL1 and ACTLR_EL2 are added.
  • Description for RGSR_EL1.TAG, bits [3:0] when GCR_EL1.RAND=1 is corrected to read "IMPLEMENTATION DEFINED".
  • External accessors for PMSSCR_EL1 are corrected from RO to RW when constraints allow.

Many simple clarifications and corrections are also present, but are too small to be listed here. Some minor formatting changes are suppressed and not highlighted in the diff output.

In keeping with the pseudocode language requirements, accessibility that operates on bit ranges within registers in the form REG<start:end> is uniformly replaced with REG<start+:width>.

Known issues

All issues identified in the below list will be fixed in a future release.

  • Accessibility for RAS registers will be relaxed to remove traps when there is a minimal RASv2 implementation.
  • The description of PMSIDR_EL1.FDS will be updated to say the feature FEAT_SPE_FDS is not mandatory if PMSIDR_EL1.LDS is 0.
  • Corresponding SET and CLR registers will be mapped to each other.
  • The UCI field in SCTLR_EL1 and SCTLR_EL2 will be corrected to describe IMPLEMENTATION DEFINED behavior when the field is 0.

Potential upcoming changes

Arm is constantly exploring ways to make the architecture presentation precise and clear. Towards this, the following changes are expected in future releases:

  • Accessibility of unimplemented registers in indexed register arrays will be improved.
  • Accessibility for memory mapped accesses to registers will be improved.
  • Accessibility when there is a width mismatch between the accessor and the register will be improved.
  • Configurability details will be captured more formally to define features, acceptable feature choices, and mapping of features to ID registers.

The details of the architecture are presented in pseudocode in Architecture Specification Language (ASL). Arm is defining a new version of the Architecture Specification Language, ASL1, to improve and expand the capabilities of the language. Please see Architecture Specification Language for details on this language. Arm plans to migrate the XML to this language later this year. A preview release shall be made available prior to this migration and announced in future releases as well as linked from the above page.


Documentation

General

A description within the XML contains the following sections:

Purpose
A short description of the purpose of the register in the Arm Architecture.
Configuration
How the register is architecturally mapped onto another System register or a memory-mapped register. If the configuration of the PE affects the implementation of the register, then information about this is also included here.
Attributes
The size of the register.
Field descriptions
The register diagram and a description of the behavior of each field within the register.

Memory-mapped registers

A memory-mapped register description also contains the following sections:

Accessing ...
The address or offset of the register in the memory map, and the accessibility.

System registers

A System register description also contains an "Accessing ..." section, that includes:

  • The assembler syntax for the instructions used to access the register, and how the instruction is encoded.
  • Pseudocode that describes the execution of all instructions used to access the register, including information about traps and enables that apply upon that access.
  • For some System registers, additional text is provided which gives extra information regarding the access to the register.
  • The accessibility pseudocode for a register assumes that that register is implemented and that all features which affect its accesses are implemented. In most cases, the behavior upon access to a register is determined in part or in whole by the Exception level at which it is accessed.
